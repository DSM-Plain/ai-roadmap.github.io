## 개요

Github는 단순한 코드 저장소를 넘어 강력한 협업 플랫폼이다. 이 문서에서는 팀 프로젝트를 효과적으로 진행하는 방법을 알아볼 것이다.

## 협업 워크플로우

### 1. Centralized Workflow (중앙 집중식)
```
모든 팀원이 main 브랜치에 직접 커밋
```
- 가장 단순한 방식
- 소규모 팀이나 개인 프로젝트에 적합
- 충돌 가능성 높음

### 2. Feature Branch Workflow
```
main (안정 버전)
  ├── feature/user-auth (개발자 A)
  ├── feature/payment (개발자 B)
  └── feature/dashboard (개발자 C)
```
- 각 기능을 별도 브랜치에서 개발
- Pull Request로 코드 리뷰 후 병합
- 가장 널리 사용되는 방식

### 3. Gitflow Workflow
```
main (배포)
  └── develop (개발)
       ├── feature/* (기능 개발)
       ├── release/* (릴리스 준비)
       └── hotfix/* (긴급 수정)
```
- 명확한 브랜치 역할 분담
- 대규모 프로젝트에 적합
- 릴리스 관리가 체계적

### 4. Forking Workflow
```
원본 저장소 (조직)
  └── Fork → 개인 저장소
       └── Pull Request → 원본에 기여
```
- 오픈소스 프로젝트의 표준
- 권한 없이도 기여 가능
- 원본 저장소 보호

## 협업 시작하기

### 팀원 초대
```
1. 저장소 Settings
2. Collaborators
3. Add people
4. 팀원 Github ID 입력
```

### 권한 설정
- **Read**: 저장소 조회만 가능
- **Write**: 푸시 가능
- **Admin**: 모든 설정 가능
- **Maintain**: 저장소 관리 (Admin 제외)

### Organization 활용
```
조직 (Company)
  ├── 팀 (Frontend Team)
  │   └── 저장소 접근 권한
  └── 팀 (Backend Team)
      └── 저장소 접근 권한
```
- 회사/그룹 단위 관리
- 팀별 권한 설정
- 프로페셔널한 관리

## 협업 규칙 설정

### Branch Protection Rules
```
Settings → Branches → Add rule
```

**주요 규칙:**
- Require pull request before merging
  - PR 없이 직접 푸시 금지

- Require approvals
  - 최소 N명의 승인 필요

- Require status checks to pass
  - CI/CD 테스트 통과 필수

- Require conversation resolution
  - 모든 리뷰 코멘트 해결 필수

- Require linear history
  - 머지 커밋 방지

- Include administrators
  - 관리자에게도 규칙 적용

### CODEOWNERS 파일
```
# .github/CODEOWNERS

# 전체 코드베이스
* @team-lead

# Frontend 디렉토리
/frontend/ @frontend-team

# API 디렉토리
/api/ @backend-team @senior-dev

# 설정 파일
*.config.js @devops-team
```
- 파일별 책임자 지정
- 자동으로 리뷰어 할당
- 코드 소유권 명확화

## 효과적인 코드 리뷰

### 리뷰어 관점

**체크리스트:**
```markdown
- [ ] 코드가 요구사항을 충족하는가?
- [ ] 버그나 에러 가능성은 없는가?
- [ ] 테스트가 충분한가?
- [ ] 성능 이슈는 없는가?
- [ ] 보안 취약점은 없는가?
- [ ] 코드 스타일이 일관적인가?
- [ ] 문서화가 필요한가?
```

**좋은 리뷰 코멘트:**
```markdown
# 구체적인 피드백
❌ "이 부분이 이상해요"
✅ "이 함수에서 null 체크가 필요할 것 같습니다.
   user가 undefined일 때 에러가 발생할 수 있습니다."

# 제안과 함께
✅ "이 로직을 useMemo로 감싸면 불필요한 재계산을 방지할 수 있습니다."

# 질문하기
✅ "이 접근 방식을 선택한 이유가 궁금합니다.
   대안으로 X 방법도 고려해보셨나요?"

# 긍정적 피드백도
✅ "깔끔한 에러 핸들링이네요! 👍"
```

### PR 작성자 관점

**리뷰 받기 전:**
```bash
# 셀프 리뷰
1. 본인이 먼저 diff 확인
2. 불필요한 변경사항 제거
3. 테스트 실행 및 통과 확인
4. 린팅/포맷팅 적용
```

**피드백 대응:**
```markdown
# 건설적으로 대응
✅ "좋은 지적입니다. 수정하겠습니다."
✅ "말씀하신 부분 고민했는데, A 이유로
   현재 방식을 선택했습니다. 의견 주세요."

# 방어적이지 않기
❌ "이게 맞는데요"
❌ "원래 이렇게 하는 거예요"
```

## Github 기능 활용

### Issues로 작업 관리
```markdown
# Issue 작성 예시
## 문제 설명
사용자 로그인 시 토큰이 만료되어도 로그아웃되지 않음

## 재현 방법
1. 로그인
2. 1시간 대기
3. 페이지 새로고침
4. 여전히 로그인 상태 유지됨

## 예상 동작
토큰 만료 시 자동 로그아웃

## 환경
- 브라우저: Chrome 120
- OS: macOS 14
```

### Labels로 분류
```
bug         빨강     버그 수정
enhancement 파랑     기능 개선
documentation 초록   문서 작업
good first issue 보라 초보자용 이슈
help wanted 주황     도움 필요
```

### Milestones로 목표 설정
```
v1.0.0 Release (2025-01-31)
  ├── #23 사용자 인증 구현
  ├── #45 결제 시스템 연동
  └── #67 대시보드 UI 개선

진행률: 2/3 완료 (66%)
```

### Projects로 칸반 보드
```
To Do       In Progress    Review      Done
  │              │            │          │
Issue #23    Issue #45    PR #56     PR #12
Issue #34    PR #48                  PR #34
```

### Discussions로 소통
```
General        아이디어 공유, 질문
Announcements  공지사항
Q&A           질문과 답변
Show and tell  작업물 공유
```

## 협업 시나리오 예시

### 신규 기능 개발
```bash
# 1. 이슈 생성 (#123)
Github에서 Issue 작성: "사용자 프로필 페이지 추가"

# 2. 브랜치 생성
git checkout -b feature/user-profile

# 3. 개발 및 커밋
git add .
git commit -m "feat: 사용자 프로필 UI 추가"

# 4. 푸시
git push origin feature/user-profile

# 5. PR 생성
Title: feat: 사용자 프로필 페이지 추가
Description: Closes #123

# 6. 리뷰 요청
Reviewers: @senior-dev, @team-lead

# 7. 피드백 반영
git add .
git commit -m "refactor: 리뷰 피드백 반영"
git push

# 8. 승인 후 머지
Merge pull request #124

# 9. 브랜치 삭제
git branch -d feature/user-profile
git push origin --delete feature/user-profile
```

### 버그 수정
```bash
# 1. 버그 이슈 확인 (#234)

# 2. Hotfix 브랜치 생성
git checkout -b hotfix/login-bug

# 3. 수정 및 테스트
git commit -m "fix: 토큰 만료 처리 로직 수정"

# 4. PR 생성 (긴급)
Label: bug, priority: high

# 5. 빠른 리뷰 및 머지

# 6. 배포 및 모니터링
```

## 커뮤니케이션 팁

### 비동기 협업
- 상세한 PR 설명 작성
- 스크린샷/영상 첨부
- 시간대 고려한 리뷰 요청

### 명확한 커뮤니케이션
```markdown
# Issue/PR 템플릿 활용
## 변경 사항
- 무엇을 변경했는지

## 이유
- 왜 변경했는지

## 테스트
- 어떻게 확인했는지

## 스크린샷
- 시각적 자료
```

### 멘션 활용
```markdown
@username 이 부분 확인 부탁드립니다
@team 전체 팀원에게 알림
```

## 협업 도구 연동

### CI/CD 통합
```yaml
# .github/workflows/pr-check.yml
name: PR Checks
on: [pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - run: npm install
      - run: npm test
      - run: npm run lint
```

### 슬랙/디스코드 연동
- PR 알림 자동 전송
- 이슈 업데이트 공유
- 배포 알림

### 프로젝트 관리 도구
- Jira 연동
- Notion 연동
- Linear 연동

## 협업 모범 사례

### DO (해야 할 것)
- 자주, 작은 단위로 커밋
- 명확한 커밋 메시지 작성
- 코드 리뷰에 적극 참여
- 문서화 습관화
- 테스트 작성

### DON'T (하지 말아야 할 것)
- 직접 main 브랜치에 푸시
- 거대한 PR 생성
- 리뷰 없이 머지
- 불필요한 파일 커밋
- 강제 푸시 (force push)

## 결론

효과적인 Github 협업의 핵심:
1. **명확한 워크플로우** 정립
2. **적극적인 코드 리뷰** 문화
3. **체계적인 이슈 관리**
4. **원활한 커뮤니케이션**

좋은 협업 습관이 좋은 코드를 만든다.
